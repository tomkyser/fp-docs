## Problem Statement

Components in `themes/foreign-policy-2017/components/` receive data via a `$locals` array passed through `\ForeignPolicy\Helpers\Templates\get_template_part()`. There is **no systematic documentation** of what keys each component expects, which are required vs. optional, what types they should be, or what defaults are applied. Developers must read source files and trace usage manually.

## Research Findings

## Architecture Decisions

### 1. Schema Location: Docs-First, Inline-on-Demand

**Primary output** — docs files only (no theme file modifications):
- `docs/05-components/_locals-shapes.md` — shared shape definitions
- `## Locals Contracts` sections added to existing component docs in `docs/05-components/`
- Contracts generated by analyzing source code, NOT by requiring PHPDoc blocks to exist first

**Optional inline annotation** — on-demand via command:
- A `/docs-locals annotate <file|dir>` command adds `@locals` PHPDoc blocks to source files
- Run per-file when you're already editing that component, or per-directory when ready
- Never required for the docs to be accurate — docs are generated from code analysis directly

**Rationale**: This avoids a 447-file commit while still producing complete contract documentation. Inline PHPDoc blocks can be added incrementally over time as files are naturally touched.

### 2. PHPDoc `@locals` Format Specification (for inline use)

```
@locals {
  key_name:  type  — Required|Optional. Description. [Default: value.]
}
```

Rules:
- One key per line, indented 2 spaces
- Colon separates key from type
- Em dash separates type from description
- "Required." or "Optional." is first word of description
- Default value stated for optional keys
- Nested keys use dot notation: `category_label.name: string`
- Array-indexed locals documented as: `[0]: array — Required. First article config.`
- HTMX components use `@controller` instead of `@locals` to denote `$cmp` pattern

### 3. HTMX Components — Separate Treatment

HTMX components (13 files in `components/htmx/`) use a `$cmp` controller object, not `$locals`. These get a `@controller` PHPDoc block documenting the controller class and the methods/state keys the template uses:

```php
<?php
/**
 * Newsletter opt-out partial
 *
 * @controller FP_Htmx_Newsletters_Controller
 * @state {
 *   push_url:  string  — URL for hx-push-url.
 * }
 * @methods {
 *   sailthru_user_is_logged_in(): bool
 *   is_sailthru_opted_out(): bool
 *   get_notifications(): array
 *   esc_target(): string
 *   esc_endpoint(): string
 *   esc_state(): string
 * }
 */
```

### 5. Integer-Indexed `$locals` — Special Handling

24 files use `$locals[0]`, `$locals[1]` instead of named keys (primarily home blocks and channel blocks). These represent a different contract pattern — the parent passes an ordered tuple of article configs.

**Documentation approach**: Document these as positional parameters with their inner structure:

```
@locals {
  [0]:  array  — Required. Primary article config (see §Article Config Shape).
  [1]:  array  — Optional. Secondary article config.
}
```

Define shared shapes (like "Article Config") once in a central reference, then reference them from individual component contracts.

### 6. Shared Shapes — Central Registry

Many components receive the same key structures (post configs, zone configs, ad configs). Rather than duplicating, define canonical shapes in a new file `docs/05-components/_locals-shapes.md`:

```markdown
# Shared $locals Shapes

## Article Config
| Key | Type | Req? | Default | Description |
|-----|------|------|---------|-------------|
| post_ID | int | Yes | — | Post ID |
| attachment_ID | int | No | from post meta | Featured image attachment ID |
| hed | string | No | post title | Headline override |

| additional_classes | string | No | '' | Extra CSS classes |
```

Components reference these: `@locals-shape ArticleConfig` instead of re-listing every key.

### 7. Contract Extraction Strategy

Since PHPDoc blocks won't exist in source files initially, the extraction engine must work from **code analysis**:

1. **Find `$locals` initialization**: Look for `get_template_part_locals()` call
2. **Extract key access**: Find all `$locals['key']` patterns in the file
3. **Classify required vs optional**:
   - Unguarded access (`$x = $locals['key']`) → Required
   - Guarded access (`$locals['key'] ?? default`, `isset($locals['key'])`, `!empty($locals['key'])`) → Optional
4. **Infer types**: From usage context (`intval()` → int, `esc_url()` → string, boolean comparisons → bool)
5. **Extract defaults**: From `??` right-hand side, ternary fallbacks, or conditional assignments
6. **Detect shared shapes**: When a component's keys match a known shape (e.g., ArticleConfig), reference it

If a `@locals` PHPDoc block IS present, prefer it over code analysis (it may document intent beyond what code analysis can infer).

### 8. WP-CLI Extraction Tool — 100% Accuracy Justification

A WP-CLI command using PHP's `token_get_all()` provides **authoritative static analysis** — the PHP tokenizer parses files identically to the PHP runtime, eliminating regex edge cases (comments, heredocs, multiline expressions, strings-within-strings).

**Why 100% accuracy is achievable for this codebase**:

Verified via exhaustive grep of all 447 component files:
- **Zero** `$locals[$variable]` — no dynamic string key access exists
- **Zero** `extract($locals)` — never used
- **Zero** string-concatenated keys — all keys are compile-time string or integer literals
- Every access is one of two static patterns: `$locals['string_literal']` or `$locals[integer_literal]`

Since every key is a constant known at parse time, `token_get_all()` can identify every single key access with complete accuracy. This makes the CLI tool the authoritative validation layer — it can run in CI or pre-commit to guarantee contracts never drift from actual code usage.

**Role in the system**: The CLI tool is the **ground truth engine**. The docs system (Claude Code) uses it for extraction and validation. The `/docs-locals` command calls it under the hood. Developers can also run it directly for quick checks.

### Phase 1: WP-CLI Extraction & Validation Tool
**Effort**: Medium
**Files**: 1 new, 1 modified

Build `wp fp-locals` — a WP-CLI command that uses PHP's `token_get_all()` to extract and validate `$locals` contracts with 100% accuracy.

| # | Task | Description |
|---|------|-------------|
| 1.1 | Create `class-locals-cli.php` with base command registration | `inc/cli/class-locals-cli.php` (new) |
| 1.2 | Implement `wp fp-locals extract <file\|dir> [--format=table\|json\|markdown]` | Tokenize PHP files, extract all `$locals['key']` and `$locals[n]` access, classify required/optional, infer types and defaults, output contract |
| 1.3 | Implement `wp fp-locals validate <file\|dir>` | Compare `@locals` PHPDoc block (if present) against actual code usage; report: undocumented keys, documented-but-unused keys, type mismatches |
| 1.4 | Implement `wp fp-locals cross-ref <file\|dir>` | Find all callers of the target component(s) across the codebase, extract the `$locals` arrays they pass, compare passed keys vs consumed keys; report mismatches |
| 1.5 | Implement `wp fp-locals coverage [--format=table\|json]` | Scan all 447 component files, report which have `@locals` PHPDoc blocks vs which don't, percentage coverage |
| 1.6 | Register CLI command in autoloader / CLI bootstrap | `inc/cli/` loader or `functions.php` CLI section |

**Technical approach**:
- **Tokenizer**: `token_get_all()` for PHP parsing (no external dependencies, VIP-compatible)
- **Key extraction**: Walk token stream, match `T_VARIABLE($locals)` → `[` → `T_CONSTANT_ENCAPSED_STRING` or `T_LNUMBER` → `]`
- **Required/optional classification**: Detect surrounding context — `isset()`, `!empty()`, `??` operator → optional; bare access → required
- **Type inference**: Look at wrapping functions (`intval()` → int, `esc_url()` → string, `esc_attr()` → string, boolean comparison → bool)
- **Default extraction**: Parse `??` right-hand operand or ternary fallback value
- **PHPDoc parsing**: Regex `/@locals\s*\{([^}]+)\}/s` to extract existing blocks for validation
- **Cross-ref**: Grep for `get_template_part('components/{slug}'` and `foreignpolicy_get_template_part('components/{slug}'`, then tokenize the caller to extract the `$locals` array literal being passed

**Output example** (`wp fp-locals extract components/post/excerpt-content--image.php`):
```
+---------------------+--------+----------+---------+----------------------------+
| Key                 | Type   | Required | Default | Context                    |
+---------------------+--------+----------+---------+----------------------------+
| post_ID             | int    | Yes      | —       | intval(), early return     |
| link                | string | No       | ''      | !empty() guard             |
| title               | string | No       | ''      | !empty() guard             |
| excerpt             | string | No       | ''      | !empty() guard             |
| additional_classes  | string | No       | ''      | ?? operator                |
+---------------------+--------+----------+---------+----------------------------+
```

**Milestone**: CLI tool can extract, validate, cross-reference, and report coverage for any component file or directory. This becomes the ground-truth engine for all subsequent phases.

## Execution Notes

### Recommended Execution Order
Phases 0 → 1 → 2 → 3 → 4 → 5 → 6 → 7

Phase 1 (CLI tool) must complete before Phase 2, since the docs generation relies on CLI output as ground truth. Phases 2 and 3 can partially overlap (cross-referencing can begin for directories whose contracts are already extracted). Phase 5 (on-demand command) can be developed in parallel with Phase 4.

### What This Plan Does NOT Include
- Bulk modification of theme source files (no mass PHPDoc injection)
- Refactoring components to standardize their `$locals` patterns
- Adding runtime type checking or validation to components
- Modifying the template-part helper API itself
- Deprecating the legacy `foreignpolicy_get_template_part()` API

### Incremental Inline Annotation Strategy
Over time, as developers touch component files for other reasons, they can run `/docs-locals annotate <file>` to add the `@locals` PHPDoc block. The annotation command uses `wp fp-locals extract` under the hood, guaranteeing the PHPDoc block matches actual code usage. This creates a natural, low-risk path to eventual full inline coverage without requiring a single massive commit. `wp fp-locals coverage` tracks progress toward full inline coverage.

### Code Analysis Accuracy
The WP-CLI tool achieves **100% extraction accuracy** for this codebase (see §8 for justification). All 447 component files use only static string-literal or integer-literal keys — no dynamic key access, no `extract()`, no variable indices. The `token_get_all()` tokenizer resolves every key deterministically.

This eliminates the primary risk from the original plan (regex-based analysis missing edge cases). The CLI tool is the authoritative source; docs and inline annotations are derived from it.

### Parallelization Opportunities
- Phase 2 subtasks (2.1–2.5) can run in parallel across component directories
- Phase 4 subtasks are independent by directory
- Phase 5 can start after Phase 0 and run in parallel with Phase 4
